Programming Languages

What is programming, really?

A computer only understands one thing: numbers that mean "add these", "store that", "jump there". Writing those numbers by hand is how it was done in the 1940s, and it was miserable. A programming language is the fix — you write something a human can read, and a translator turns it into those numbers for you.

Two kinds of translator, and it's the single biggest split between languages. A compiler translates the whole program up front, then you run the result — fast, but you wait to build. An interpreter translates line by line as it runs — instant feedback, slower execution.

So why hundreds of languages instead of one? Because every language is a set of trade-offs: run faster or write faster, catch mistakes early or stay flexible, control memory yourself or let the machine handle it. A language that's perfect for a game engine is a poor fit for a data analysis script. Pick by what you're building — that's what this page is for.

The three questions

How languages actually differ

Almost every language decision comes down to these three. The tags on each language below use this vocabulary.

When is it translated?
CompiledvsInterpreted
Compiled languages translate ahead of time and run at full machine speed. Interpreted ones translate as they go — you skip the build step and get instant feedback, at a runtime cost.
C++, Java, Go  ·  Python, JavaScript, Ruby
When are type errors caught?
StaticvsDynamic
Static typing checks that you never put a string where a number belongs before the program runs — more upfront ceremony, fewer 3am surprises. Dynamic typing checks while running: faster to write, easier to break.
Java, C++, Rust, TypeScript  ·  Python, JavaScript, Ruby
Who cleans up memory?
ManualvsGarbage-collected
Manual memory means you decide exactly when every byte is freed — maximum control, and the origin of most famous security bugs. A garbage collector reclaims memory for you, costing unpredictable pauses.
C, C++  ·  Java, Python, Go, JavaScript
Side by side

The same program in four languages

A counter class, a loop, and a print — the same logic in each. Watch what changes: how types are declared, how blocks are marked, how much ceremony wraps the idea.

Python — indentation marks blocks, no type declarations
class Counter:
    def __init__(self):
        self.count = 0

    def bump(self):
        self.count += 1
        return self.count

c = Counter()
for _ in range(3):
    print(c.bump())
JavaScript — braces mark blocks, still no types
class Counter {
  constructor() {
    this.count = 0;
  }

  bump() {
    return ++this.count;
  }
}

const c = new Counter();
for (let i = 0; i < 3; i++) {
  console.log(c.bump());
}
Java — every type declared, everything inside a class
class Counter {
    private int count = 0;

    public int bump() {
        return ++count;
    }
}

public class Main {
    public static void main(String[] args) {
        Counter c = new Counter();
        for (int i = 0; i < 3; i++) {
            System.out.println(c.bump());
        }
    }
}
C++ — types declared, plus an explicit include and entry point
#include <iostream>

class Counter {
    int count = 0;
public:
    int bump() { return ++count; }
};

int main() {
    Counter c;
    for (int i = 0; i < 3; i++) {
        std::cout << c.bump() << std::endl;
    }
}

Same idea, four dialects. Python says it in 11 lines with no type names; Java needs a class wrapper and a declared type on everything. Neither is "better" — the ceremony Java charges you buys errors caught before the program ever runs.
